# 核心算法修改计划

时间标签：2026-06-17 16:28 UTC

## 背景

当前项目的主要目标是以 SmolVLA 作为基础模型完成 LIBERO 四类任务的训练、验证和探索，同时保留后续切换到其他 VLA 或 action-chunk policy 的能力。近期排查发现，之前训练完成后的 rollout 视频中机械臂基本不动，且保存视频的 image shape 也存在问题。视频 shape 问题已经独立纳入运行基础设施修复；核心算法问题集中在当前 PA-TCS 训练逻辑和预期算法不一致。

## 当前问题

现有实现更接近“固定时间 BC chunk + PA-TCS 正则”，还不是真正的动态 PA-TCS。

当前训练逻辑：

- DataLoader 预先切好 `(obs_t, action[t:t+H])`。
- `rho_start` 按 demo 时间固定计算。
- 模型输出的当前 chunk 不影响下一个 chunk 的监督位置。
- 下一个训练样本仍来自预切好的固定时间片。
- PA-TCS 只是在固定 chunk 上附加 tube/event loss。

这个逻辑的问题是，模型仍然被迫拟合单条 demo 的固定动作片段。如果 event loss 尺度过大、事件半径过小、BC 和 PA-TCS loss 没有合理归一化，模型可能选择近零动作作为低风险解，最终表现为机械臂不动。

## 目标算法

目标 PA-TCS 逻辑应该允许模型在监督轨迹云中探索连续动作的位置和速度，而不是强制逐点拟合单条 demo 的固定 action chunk。

理想训练流程：

1. 一个训练 worker 接管一条 trajectory。
2. worker 从轨迹开头的 observation 开始。
3. 模型根据当前 observation 输出一个 action chunk。
4. 根据预测 action chunk 的末端路径或 ee path，在 PA-TCS cloud / anchor / monotonic matcher 中估计新的进度 `rho_next`。
5. 用 `rho_next` 动态选择下一段监督 observation / action reference。
6. 重复执行，直到监督轨迹覆盖到 terminal event。
7. 当前 trajectory 完成后，worker 再领取下一条 trajectory。

这个逻辑本质上不再是普通 map-style Dataset，而更像训练循环内部的 `TrajectorySupervisor` 或 `IterableDataset`。batch 中的每个 item 可以视为一个 worker state。

## Loss 设计

核心 loss 需要从单一固定 BC 扩展为进度感知的组合目标。

计划包含：

- `tube loss`：预测路径应处于当前阶段的概率云或 tube 内，允许合理速度和位置偏差。
- `event loss`：关键事件点收紧，例如抓取、接触、放置等阶段边界。
- `monotonic progress loss`：预测进度必须单调前进。
- `stall penalty`：惩罚进度停滞，防止模型学到不动。
- `backward penalty`：惩罚进度倒退。
- `BC anchor loss`：保留弱锚点，辅助稳定训练，但不能重新退化成固定 BC。
- `smoothness loss`：约束 chunk 内抖动和 chunk 间边界跳变。

需要重点修正的尺度问题：

- `event_radius` 不能过小，之前 `1e-4` 量级不合理，应先在 `0.01 ~ 0.03` 范围做 ablation。
- `event loss` 需要 normalize 或 clip，不能压过 BC / tube loss。
- 训练日志必须记录 action norm、xyz displacement、repeated-action rate、all-zero rate。

## 离线训练限制

离线训练没有真实环境状态转移。模型预测一个新末端位置后，我们没有对应的新相机 observation。

可选近似：

1. 动态进度，demo observation 对齐。
   根据预测路径匹配到参考 demo 的新进度，然后取该进度附近的 demo observation 作为下一步输入。这是最接近当前目标、也最容易先落地的方案。

2. trajectory-level alignment。
   不真的滚动 observation，而是在较长窗口内输出多个 chunks，用 monotonic alignment / Soft-DTW 对齐 cloud。这更稳定，但对 SmolVLA 训练接口改动更大。

当前建议先实现方案 1，再评估是否需要方案 2。

## 分阶段计划

### Phase 1：SmolVLA 基线

目标：确认模型、数据、eval、视频链路没有基础错误。

任务：

- 抽象 policy backend 接口，保留 ActionMLP 作为 smoke backend。
- 接入 SmolVLA adapter。
- 使用 LeRobot / LIBERO 数据跑 BC-only。
- 验证 rollout 中机械臂至少能动。
- 确认训练输入和 eval observation schema 一致。
- 固定视频保存 shape、dtype、fps。

验收：

- BC-only rollout action 非全零。
- 视频 shape 正确。
- action health 指标正常。

### Phase 2：pi0.5 成功轨迹收集

目标：用更强的 pi0.5 LIBERO checkpoint 收集连续成功轨迹，作为新的 PA-TCS artifact 来源。

当前已完成：

- OpenPI 独立 `pi` 环境。
- New-IL 主环境负责 LIBERO、队列、视频、NPZ 保存。
- OpenPI 只作为 websocket policy server。
- Mem-style 并行队列和 deficit scheduling。
- 默认 8 GPU，每卡 6 server + 6 worker。
- 默认任务池包含 `libero_spatial,libero_object,libero_goal,libero_10`。
- 默认不收集 `libero_90`。
- 每个 suite/task 目标 100 条成功轨迹，总目标 4000 条。

启动命令：

```bash
cd /data/L202500340/New-IL && bash scripts/run-openpi-libero-8gpu.sh
```

验收：

- 成功 rollout NPZ 中包含 `observation.images.image`、`observation.images.image2`、`observation.state`、`actions`、`language`、`success`。
- action health 显示非 all-zero。
- 视频尺寸稳定。

### Phase 3：从 rollout NPZ 重建 PA-TCS artifact

目标：不再只依赖原始 demo，而是从成功 rollout 中重建任务轨迹云。

当前已完成：

- `new-il-build-patcs-artifacts --source-format rollout_npz` 支持从 Mem-style rollout NPZ 构建 artifact。

后续任务：

- 为四个 suite 的每个 task 建 artifact。
- 按 task 记录 demo count、stage count、phase count、event mask。
- 对 event radius、margin、num_phase 做小规模 ablation。

验收：

- 每个 task artifact 可加载。
- artifact 中 event mask、anchor、phase_points shape 正确。
- 随机预测 loss 高于 anchor/reference 预测。

### Phase 4：静态 PA-TCS ablation

目标：先在固定 chunk 训练里验证 loss 尺度，避免直接进入动态训练后难以定位问题。

任务：

- BC-only。
- BC + tube。
- BC + tube + event。
- BC + tube + event + stall/backward proxy。
- 记录每组 action norm、xyz displacement、success rate、视频。

验收：

- 加 PA-TCS 后不能导致 action norm 塌缩。
- event loss 不应长期主导 total loss。
- rollout 视频中机械臂不再完全静止。

### Phase 5：动态 Trajectory Worker PA-TCS

目标：实现真正的动态进度训练。

核心状态：

- `task_id`
- `demo_id`
- `stage`
- `rho`
- `obs_index`
- `previous_chunk`
- `done`

核心流程：

1. worker 读取当前 observation。
2. policy 输出 action chunk。
3. 积分或解析得到 predicted ee path。
4. 使用 PA-TCS cloud / anchor / monotonic matcher 得到 `rho_next`。
5. 计算 tube/event/progress/stall/backward/smoothness loss。
6. 用 `rho_next` 选择下一个 demo observation index。
7. 当前 trajectory 到 terminal event 后重置并领取新 trajectory。

验收：

- `rho` 随训练单调推进。
- stall rate 下降。
- action norm 不塌缩。
- 相比静态 PA-TCS，rollout 成功率或阶段完成率提升。

## 风险

- 离线动态推进只能近似环境状态转移，可能出现 observation 和预测动作不完全一致。
- pi0.5 收集数据若分布过窄，PA-TCS cloud 可能过于保守。
- 多 server 多 worker 会带来 GPU 显存压力，需要根据实际机器调整 `SERVERS_PER_GPU`。
- SmolVLA 的 action head / normalization 需要和 LIBERO action schema 严格对齐，否则 loss 正确也可能 rollout 异常。
